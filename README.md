# Pattern_NET_Oris

Ниже кратко описано, какой паттерн реализован в каждой библиотеке классов, как он работает, какую проблему решает именно в этом коде и где его можно применять в реальной программе на C#.

В этом репозитории собраны паттерны, которые решают разные типовые задачи:
- **Adapter** — делает совместимыми разные интерфейсы;
- **Builder** — удобно собирает сложный объект;
- **Chain of Responsibility** — передаёт запрос по цепочке обработчиков;
- **Decorator** — добавляет объекту новое поведение без изменения его класса;
- **Factory Method** — выносит создание объекта в отдельную фабрику;
- **Iterator** — даёт единый способ обхода коллекции;
- **Observer** — уведомляет подписчиков об изменениях;
- **Singleton** — гарантирует один экземпляр;
- **Strategy** — позволяет подменять алгоритм;
- **Abstract Factory** — создаёт семейства связанных объектов.

## Pattern.Adapter

### Что за паттерн
**Adapter (Адаптер)**.

### Как работает
Адаптер позволяет объектам с несовместимыми интерфейсами работать вместе. В этом проекте `SocketToUsbAdapter` принимает `EuropeanSocket`, но наружу предоставляет интерфейс `IUsbCharger`.

### Что решается в этом коде
В коде есть обычная европейская розетка `EuropeanSocket`, которая умеет только `Provide220Volts()`, и есть клиент, который ожидает зарядку через `IUsbCharger`. Напрямую они несовместимы. Адаптер берёт напряжение из розетки, «преобразует» его и даёт понятный клиенту способ `ChargeWithUsb()`.

### Почему здесь подходит именно этот подход
Потому что исходный класс розетки менять не нужно. Мы просто добавляем промежуточный слой, который переводит один интерфейс в другой.

### Как применять в своей программе на C#
Используй Adapter, когда:
- нужно подключить старый класс к новому коду;
- сторонняя библиотека имеет неудобный интерфейс;
- ты хочешь обернуть внешний API под свой понятный интерфейс.

### Пример использования
```csharp
var socket = new EuropeanSocket();
IUsbCharger charger = new SocketToUsbAdapter(socket);

charger.ChargeWithUsb();
```

## Pattern.Builder

### Что за паттерн
**Builder (Строитель)**.

### Как работает
Builder пошагово собирает сложный объект. В проекте `PizzaBuilder` поэтапно задаёт свойства пиццы, а `PizzaDirector` хранит готовые сценарии сборки: например, `MakeMargherita()` и `MakePepperoni()`.

### Что решается в этом коде
Объект `Pizza` может иметь много параметров: тесто, соус, сыр, мясо, грибы. Если создавать его через большой конструктор, код станет неудобным и нечитаемым. Builder позволяет собирать объект по шагам.

### Почему здесь подходит именно этот подход
Потому что у пиццы много необязательных частей, и не каждая пицца должна содержать все ингредиенты. Builder делает создание объекта понятным и гибким.

### Как применять в своей программе на C#
Используй Builder, когда:
- объект имеет много параметров;
- есть обязательные и необязательные поля;
- нужно создавать разные конфигурации одного типа объектов.

### Пример использования
```csharp
var director = new PizzaDirector();
var builder = new PizzaBuilder();

Pizza margherita = director.MakeMargherita(builder);
margherita.Show();
```

## Pattern.ChainOfResponsibilites

### Что за паттерн
**Chain of Responsibility (Цепочка обязанностей)**.

### Как работает
Запрос передаётся по цепочке обработчиков, пока кто-то из них не сможет его обработать. В проекте это `BasicSupport`, `ExpertSupport`, `ManagerSupport`.

### Что решается в этом коде
Есть обращения разной сложности, которые определяются по `severity`. Простые вопросы решает базовая поддержка, более сложные — эксперт, ещё более сложные — менеджер. Так клиент не выбирает сам, кто должен решать проблему.

### Почему здесь подходит именно этот подход
Потому что логика маршрутизации запроса естественно распределяется между несколькими уровнями обработки. Это убирает большие `if/else` или `switch` в одном месте.

### Как применять в своей программе на C#
Используй Chain of Responsibility, когда:
- есть несколько уровней обработки;
- запрос может быть обработан разными объектами;
- важно гибко перестраивать порядок обработки.

### Пример использования
```csharp
var basic = new BasicSupport();
var expert = new ExpertSupport();
var manager = new ManagerSupport();

basic.SetNext(expert);
expert.SetNext(manager);

basic.HandleRequest("Не работает кнопка входа", 2);
basic.HandleRequest("Сервер периодически падает", 4);
basic.HandleRequest("Критическая авария в системе", 5);
```

## Pattern.Decorator

### Что за паттерн
**Decorator (Декоратор)**.

### Как работает
Декоратор оборачивает объект с тем же интерфейсом и добавляет поведение до или после основного вызова. В проекте все уведомления работают через `INotifier`, а `LoggingNotifier` добавляет логирование перед отправкой.

### Что решается в этом коде
Есть базовая отправка уведомления, например `EmailNotifier` или `SmsNotifier`. Нужно добавить дополнительную функциональность — логирование — не меняя код самих уведомителей. `LoggingNotifier` решает это, оборачивая любой объект `INotifier`.

### Почему здесь подходит именно этот подход
Потому что логирование — это дополнительная обязанность, а не новая основная сущность. Декоратор позволяет расширить поведение без изменения базовых классов.

### Как применять в своей программе на C#
Используй Decorator, когда:
- надо добавить логирование, кеширование, авторизацию, метрики;
- не хочется раздувать наследование;
- нужно комбинировать несколько дополнительных функций.

### Пример использования
```csharp
INotifier notifier = new EmailNotifier();
INotifier loggingNotifier = new LoggingNotifier(notifier);

loggingNotifier.Send("Заказ оформлен");
```

## Pattern.FabricMethod

### Что за паттерн
**Factory Method (Фабричный метод)**.

### Как работает
Фабричный метод выносит создание объектов в отдельные классы-фабрики. В проекте абстрактный `TransportFactory` определяет метод `CreateTransport()`, а конкретные фабрики создают `Car`, `Bicycle` или `Airplane`.

### Что решается в этом коде
Клиенту нужен транспорт, но он не должен сам решать, какой именно класс создавать через `new`. Фабрика скрывает создание объекта и позволяет централизованно управлять этим процессом.

### Почему здесь подходит именно этот подход
Потому что создание транспорта — отдельная ответственность. Код, который заказывает доставку, работает через общий интерфейс `ITransport`, а конкретный тип выбирается фабрикой.

### Как применять в своей программе на C#
Используй Factory Method, когда:
- нужно скрыть создание объектов;
- тип объекта зависит от условий;
- хочется снизить связанность между клиентом и конкретными классами.

### Пример использования
```csharp
TransportFactory factory = new CarFactory();
ITransport transport = factory.OrderTransport();

transport.Deliver();
```

## Pattern.Iterator

### Что за паттерн
**Iterator (Итератор)**.

### Как работает
Итератор даёт способ обходить элементы коллекции, не раскрывая её внутреннее устройство. В проекте `BookCollection` хранит список книг, а `BookIterator` последовательно возвращает их через `HasNext()` и `Next()`.

### Что решается в этом коде
Клиенту нужно пройтись по книгам, но ему не нужно знать, как именно книги хранятся внутри коллекции. Итератор отделяет логику обхода от самой коллекции.

### Почему здесь подходит именно этот подход
Потому что коллекция должна отвечать за хранение, а отдельный объект — за обход. Это делает код чище и удобнее для расширения.

### Как применять в своей программе на C#
Используй Iterator, когда:
- нужен контролируемый обход коллекции;
- не хочется раскрывать внутреннюю структуру;
- нужен единый способ обхода разных коллекций.

### Пример использования
```csharp
var books = new BookCollection();
books.AddBook("CLR via C#");
books.AddBook("Clean Code");
books.AddBook("Head First Design Patterns");

var iterator = books.CreateIterator();
while (iterator.HasNext())
{
    Console.WriteLine(iterator.Next());
}
```

## Pattern.Observer

### Что за паттерн
**Observer (Наблюдатель)**.

### Как работает
Один объект хранит список подписчиков и уведомляет их об изменениях. В проекте `NewsPublisher` публикует новость и вызывает `Update()` у всех подписчиков: `EmailSubscriber` и `SmsSubscriber`.

### Что решается в этом коде
Когда выходит новая новость, её нужно сразу отправить нескольким получателям разными способами. Без Observer издатель был бы жёстко связан с каждым типом подписчика. Здесь он работает только через интерфейс `IObserver`.

### Почему здесь подходит именно этот подход
Потому что есть естественная схема «один ко многим»: один источник событий и много слушателей. Это уменьшает связанность и упрощает добавление новых подписчиков.

### Как применять в своей программе на C#
Используй Observer, когда:
- после изменения состояния надо уведомить несколько объектов;
- подписчики могут добавляться и удаляться динамически;
- важна слабая связанность между источником и получателями.

### Пример использования
```csharp
class Program
{
    static void Main()
    {
        var publisher = new NewsPublisher();

        var alice = new EmailSubscriber("alice@example.com");
        var bob = new EmailSubscriber("bob@example.com");
        var carol = new SmsSubscriber("+123456789");

        publisher.Attach(alice);
        publisher.Attach(bob);
        publisher.Attach(carol);

        publisher.PublishNews("C# 12 вышел!");

        publisher.Detach(bob);

        publisher.PublishNews("Выходные близко!");
    }
}
```

## Pattern.Singleton

### Что за паттерн
**Singleton (Одиночка)**.

### Как работает
Singleton гарантирует, что у класса будет только один экземпляр, и даёт глобальную точку доступа к нему. В проекте `DatabaseSingleton` один раз читает строку подключения из `appsettings.json`, создаёт объект и отдаёт его через `Instance`.

### Что решается в этом коде
Для подключения к базе нужен единый объект конфигурации, который не должен создаваться много раз в разных местах программы. Singleton даёт централизованный доступ к одному экземпляру.

### Почему здесь подходит именно этот подход
Потому что строка подключения и объект доступа к ней логически общие для всего приложения. Нет смысла создавать множество одинаковых экземпляров.

### Как применять в своей программе на C#
Используй Singleton осторожно, когда:
- объект действительно должен быть один на всё приложение;
- нужен единый доступ к конфигурации, логгеру или сервису состояния.

> На практике в современных приложениях C# часто вместо Singleton удобнее использовать DI-контейнер с lifetime `Singleton`.

### Пример использования
```csharp
var db1 = DatabaseSingleton.Instance;
var db2 = DatabaseSingleton.Instance;

Console.WriteLine(object.ReferenceEquals(db1, db2));
```

## Pattern.Strategy

### Что за паттерн
**Strategy (Стратегия)**.

### Как работает
Стратегия выносит разные алгоритмы в отдельные классы с общим интерфейсом. В проекте `Order` хранит ссылку на `IPaymentStrategy`, а конкретная стратегия — `CreditCardPayment` или `PayPalPayment` — определяет, как именно выполняется оплата.

### Что решается в этом коде
Есть один и тот же процесс оплаты заказа, но способов оплаты несколько. Вместо большого `if/else` по типу оплаты код делится на отдельные классы, которые можно подменять.

### Почему здесь подходит именно этот подход
Потому что алгоритм оплаты меняется независимо от самого заказа. `Order` не знает деталей каждой оплаты, а просто вызывает общий метод `Pay()`.

### Как применять в своей программе на C#
Используй Strategy, когда:
- есть несколько вариантов одного алгоритма;
- нужно переключать поведение во время выполнения;
- хочется убрать ветвления по типу логики.

### Пример использования
```csharp
var order = new Order(2500m);
order.SetPaymentStrategy(new CreditCardPayment());
order.IsSetPaymentStrategy(new CreditCardPayment());
```

## Patterns.AbstractionFabricMethod

### Что за паттерн
**Abstract Factory (Абстрактная фабрика)**.

### Как работает
Абстрактная фабрика создаёт семейства связанных объектов. В проекте `IGUIFactory` создаёт сразу согласованные элементы интерфейса: `IButton` и `ICheckbox`. Конкретные фабрики `WindowsFactory` и `MacFactory` возвращают элементы в одном стиле.

### Что решается в этом коде
Приложению нужен интерфейс под разные платформы, но элементы должны быть совместимыми между собой по стилю. Нельзя смешать `MacButton` и `WindowsCheckbox` случайно, если всё создаётся через одну фабрику.

### Почему здесь подходит именно этот подход
Потому что здесь создаётся не один объект, а целое семейство связанных объектов. Это как раз типичный случай для Abstract Factory.

### Как применять в своей программе на C#
Используй Abstract Factory, когда:
- нужно создавать группы связанных объектов;
- есть несколько семейств реализаций;
- важно сохранять совместимость объектов внутри одного семейства.

### Пример использования
```csharp
IGUIFactory factory = new WindowsFactory();
var app = new Application(factory);

app.RenderUI();
app.HandleUserAction();
```
